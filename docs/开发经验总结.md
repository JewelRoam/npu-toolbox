# NPU工具箱开发经验总结

## 项目概述

NPU工具箱是一个基于 Tauri v2 + React + TypeScript 的桌面应用程序，旨在让用户能够轻松使用本地 AI 能力，特别是利用 NPU（神经处理单元）进行加速。应用通过 PowerShell CIM/WMI 接口实现真实的硬件信息检测，采用五重探测算法最大化 NPU 识别准确率。

## 技术栈选择

### 前端
- **React 18**: 使用函数组件和 Hooks
- **TypeScript**: 类型安全，前后端结构体字段名保持 snake_case 一致
- **Tailwind CSS**: 快速样式开发
- **React Router v6**: 页面路由
- **Zustand**: 状态管理，带 persist 中间件实现设置持久化
- **Vite**: 快速开发构建

### 后端
- **Tauri v2**: Rust 桌面应用框架
- **tokio**: 异步运行时
- **serde / serde_json**: JSON 序列化/反序列化
- **log / env_logger**: 日志系统
- **dirs**: 跨平台配置目录获取

```
Frontend UI → invoke() → Rust Command → PowerShell (CIM/WMI) → Parse output → Serialize → Frontend render
```

## NPU 检测实现详解

### 五重探测算法

NPU 检测是本项目的核心功能，采用由快到慢、由精确到模糊的五重策略：

#### 1. CIM 设备枚举（最精确）
通过 `Win32_PnPEntity` 查询所有即插即用设备，匹配 NPU 相关关键词（Intel AI Boost、AMD XDNA 等），同时排除 HID/USB/DISPLAY 等误匹配设备 ID。通过设备 ID 中的 `VEN_xxxx` 判断厂商。

#### 2. CPU 型号推断（最常见场景）
多数 NPU 集成在 CPU 中（如 Intel Core Ultra、AMD Ryzen AI）。通过 `Win32_Processor` 获取 CPU 名称，用正则匹配已知 NPU 型号。

#### 3. 注册表扫描（覆盖面广）
扫描 `HKLM:\SYSTEM\CurrentControlSet` 下的多个注册表路径，查找 DriverDesc/Service/DeviceDesc 中包含 AI/NPU/Neural 的条目。

#### 4. 驱动文件检测（离线可用）
直接检查 `System32\drivers` 和 `SysWOW64\drivers` 目录下已知 NPU 驱动文件名（iguard.sys、IntelNpuDriver.sys 等）。

#### 5. 设备 ID 匹配（最可靠）
通过 `Get-PnpDevice` 匹配已知 NPU PCI 设备 ID（如 `VEN_8086&DEV_7D55` 为 Meteor Lake NPU）。

### 设计决策
- 所有方法串行执行，首个成功即返回（early return）
- PowerShell 调用统一使用 `-NoProfile` 参数避免加载用户配置导致启动缓慢
- 每种方法独立返回 `Result<Option<NPUStatus>, String>`，互不影响

## 开发过程中的挑战与解决方案

### 1. pip install 参数格式错误

**问题**: 通过 `pip install "openvino openvino-genai"` 安装时，pip 将空格分隔的两个包名当作一个包名解析，报错 `Invalid requirement: 'openvino openvino-genai': Expected end or semicolon`。

**解决方案**: 将多个包拆分为独立的命令行参数，而非用空格拼接在一个字符串中：
```rust
// Wrong: pip receives one arg "openvino openvino-genai"
let args = ["install", "openvino openvino-genai"];

// Correct: pip receives two separate args
let args = ["install", "openvino", "openvino-genai"];
```
关键原则：每个 pip 包名必须是独立的 `Command::arg()` 调用，不能合并为一个字符串。

### 2. Tauri v2 配置变化

**问题**: Tauri v2 与 v1 配置格式差异较大，权限系统完全重写。

**解决方案**:
- 创建 `src-tauri/capabilities/default.json` 配置窗口权限
- 使用 `core:default` 授予基础 IPC 权限（包含 `invoke` 调用）
- 使用 `shell:allow-open` 授予打开外部链接权限
- 在 `tauri.conf.json` 中使用 `app.windows` 替代旧的顶层 `tauri.windows` 配置
- identifier 使用 `com.nputoolbox.desktop` 符合 Tauri v2 规范

### 3. Windows 构建环境

**问题**: Rust 链接阶段需要 MSVC Build Tools 的 `link.exe`，仅装 Rustup 不够。

**解决方案**:
- 安装 [Visual Studio Build Tools](https://visualstudio.microsoft.com/visual-cpp-build-tools/)
- 勾选「C++ 桌面开发」工作负载
- 确保 `VC Tools` 和 `Windows SDK` 都已安装
- 前端构建（`npm run build`）和 Rust 编译（`cargo build`）可独立进行，只有最终链接需要 MSVC

### 4. 硬件信息获取

**问题**: Rust 没有直接的 Windows 硬件信息 API，第三方 crate（如 system-info）功能有限。

**解决方案**: 通过 PowerShell CIM 命令获取硬件信息，使用结构化文本输出格式（`KEY=VALUE`）便于 Rust 解析：
```rust
fn run_powershell(script: &str) -> Result<String, String> {
    let output = Command::new("powershell")
        .args(["-NoProfile", "-Command", script])
        .output()
        .map_err(|e| format!("PowerShell execution failed: {}", e))?;
    Ok(String::from_utf8_lossy(&output.stdout).to_string())
}
```

### 5. 前后端类型一致性

**问题**: Rust 使用 `snake_case` 序列化，前端 TypeScript 接口需要匹配。

**解决方案**: 前端接口也统一使用 `snake_case`，不使用 `#[serde(rename_all = "camelCase")]`：
```typescript
interface NPUStatus {
  has_npu: boolean    // matches Rust: has_npu
  driver_version: string  // matches Rust: driver_version
}
```

### 6. Zustand Store 的 persist 中间件

**问题**: 使用 `persist` 中间件时，`partialize` 配置不当会导致不需要持久化的状态（如 loading）被缓存。

**解决方案**: 只持久化 settings 和 tools：
```typescript
persist(
  (set, get) => ({ /* ... */ }),
  { name: 'npu-toolbox-storage', partialize: (state) => ({ settings: state.settings, tools: state.tools }) }
)
```

### 7. PowerShell 启动速度

**问题**: 默认 PowerShell 启动会加载用户 profile，首次调用可能需要 2-3 秒。

**解决方案**: 所有 PowerShell 调用统一添加 `-NoProfile` 参数，启动时间降至 200-500ms。

### 8. Python venv 自动化管理

**问题**: 用户可能未安装 Python、已安装但版本不兼容、或已有冲突的虚拟环境。

**解决方案**: 
- 使用 `%APPDATA%\NPUToolbox\venv` 作为固定 venv 路径，与系统 Python 隔离
- `ensure_venv()` 在每次操作前检查并自动创建 venv（`python -m venv`）
- 所有 pip 操作通过 `venv\Scripts\python.exe -m pip` 执行，确保依赖安装在正确位置
- 通过 Tauri 事件 (`pip-progress`) 实时将 pip 输出推送到前端显示

### 9. Ollama 软件内一键安装

**问题**: 原方案跳转 Ollama 官网让用户自行下载安装，体验割裂且用户可能不会配置。

**解决方案**:
- 后端实现 `ollama_install` 命令，通过 GitHub API 获取最新版本下载链接
- 使用 `reqwest` 流式下载安装包，通过 `ollama-install-progress` 事件推送进度（百分比）
- 下载完成后静默执行安装程序（`/S` 静默参数）
- 安装完成后自动尝试连接 Ollama 服务

### 10. OpenVINO 安装变体设计

**问题**: OpenVINO 有 NPU、GPU、CPU 等不同使用场景，未来可能需要安装不同的包组合。

**解决方案**: 
- 定义 `OpenVINOVariant` 枚举（Npu/Gpu/Cpu），通过 `#[serde(rename_all = "lowercase")]` 支持前端字符串传参
- 每个变体的 `packages()` 方法返回不同的 pip 包列表
- 当前 Npu 和 Gpu 安装相同包，但 Gpu 标记了 `TODO` 注释预留差异化空间
- 前端安装按钮传递 variant 参数，可轻松扩展为下拉选择器

### 11. 推理测试基准化

**问题**: 单次推理延迟测试结果波动大，无法反映真实性能。

**解决方案**:
- Python 脚本执行 3 次 Warmup（避免冷启动干扰）+ 10 次 timed trial
- 后端收集所有 trial 数据，计算 avg/min/max/P50/P95 百分位统计
- 前端使用 6 列 StatBox 网格展示完整基准测试结果

### 12. Windows cmd.exe 中 git commit 的转义问题

**问题**: 在 Windows cmd.exe 中直接使用 `git commit -m "multi-line message"` 会失败，因为 cmd.exe 对引号、换行、连字符（`-`）的处理与 bash 完全不同。

**解决方案**: 使用临时文件传递 commit message，配合 `git commit -F`：
```cmd
(echo first line & echo. & echo - second line with dash) > .gm && git commit -F .gm && del .gm
```
注意：`.gm` 已添加到 `.gitignore` 防止意外提交。

## 最佳实践

### 1. 错误处理

- 使用 `Result<T, String>` 作为命令返回类型
- PowerShell 脚本内部使用 `try/catch`，输出 `ERROR=xxx` 格式错误信息
- Rust 端解析时优先检查 ERROR 行
- 前端显示友好的错误提示卡片，不直接展示原始错误

### 2. 日志规范

- 使用 `log` crate 进行日志记录
- 日志信息使用英文，便于国际化
- 合理使用 `info!`（正常流程）、`warn!`（检测失败但可继续）、`error!`（严重错误）
- 开发模式通过 `RUST_LOG=debug` 查看完整探测过程

### 3. 代码组织

```
src-tauri/
├── src/
│   ├── main.rs           # App entry + command registration
│   ├── commands.rs       # Tauri command implementations
│   └── npu_detector.rs   # Isolated NPU detection module
├── capabilities/
│   └── default.json      # Tauri v2 permissions
├── tauri.conf.json       # App config
└── Cargo.toml

src/
├── components/           # Reusable components (Layout, Sidebar, Header)
├── pages/                # Page components (Home, HardwareInfo, AIChat, etc.)
├── stores/               # Zustand state management
```

使用用户配置目录存储设置：

## 性能优化建议

1. **前端**: 使用 React.memo 减少不必要的重渲染
2. **前端**: Layout 的 PageSlot 已实现 always-mounted + `display:none` 切换，避免重复挂载开销
3. **后端**: 硬件信息获取可改为 `tokio::join!` 并行执行（当前串行，可优化）
4. **PowerShell**: `-NoProfile` 是最重要的优化，避免每次调用加载 profile
5. **缓存**: 对频繁请求的硬件数据实现缓存机制（如 5 秒内不重复检测）
6. **Python**: OpenVINO 检测脚本可用 `ov.__version__` 快速判断，无需每次都创建 Core 实例

## 已知限制

1. **GPU 使用率**: 需要 DirectX/NVAPI/ADL 等专用 API，当前返回 0.0
2. **NPU 算力**: TOPS 值基于已知型号硬编码，无法动态获取
3. **磁盘健康**: SMART 数据需要管理员权限，当前返回 100
4. **Qualcomm NPU**: Windows on ARM 平台暂未支持
5. **温度数据**: CPU 温度通过 WMI `MSApi_ThermalZoneTemperature` 获取（部分设备可用），GPU 温度通过 OpenHardwareMonitor 或 `nvidia-smi` 获取
6. **OpenVINO GPU 变体**: 当前 GPU 版本安装的包与 NPU 版相同，待 Intel 发布差异化 GPU 包后更新
7. **Ollama 静默安装**: 依赖安装程序支持 `/S` 参数，部分版本可能行为不一致

## 未来改进方向

1. **NPU 深度集成**: 通过 OpenVINO 直接调用 NPU 进行推理，提供模型加载和推理的完整流程
2. **并行探测**: 使用 `tokio::join!` 并行执行五重 NPU 检测
3. **模型管理**: 模型库页面集成下载管理，支持从 HuggingFace 直接拉取并转换为 OpenVINO IR
4. **实时监控**: 硬件状态的定时刷新（当前仅手动刷新）
5. **多语言支持**: 国际化 (i18n)
6. **主题系统**: 深色/浅色模式切换完善
7. **管理员模式**: 可选提权以获取温度、SMART 等高级信息
8. **OpenVINO GPU 变体**: 当 Intel 发布 GPU 优化包后，实现差异化的 GPU 安装路径
9. **Ollama 模型管理**: 在 AI 对话页面直接拉取/删除 Ollama 模型
